# SBOM Strategy - Supply Chain Transparency

## 1. What Exists Today

Addi's CI/CD pipeline already generates Software Bill of Materials (SBOM) artifacts for every container image. Trivy scans each image during build and produces a CycloneDX SBOM capturing all OS packages and application dependencies. These SBOMs are cryptographically attached to container images as OCI artifacts using Cosign, creating an immutable audit trail. Golden base images include pre-computed SBOMs generated by Syft, ensuring layered traceability from foundational components upward. This foundation satisfies the basic inventory requirement but lacks the queryability and automation needed for proactive vulnerability management and compliance verification.

## 2. What Dependency-Track Adds

Dependency-Track transforms static SBOMs into a queryable vulnerability intelligence platform. It indexes all component versions across services, enabling instant answers to queries like "Which of our 1000 services use libexpat?" or "Where is Log4j 2.14.0 deployed?" When new CVEs publish, Dependency-Track automatically matches them against indexed SBOMs and triggers alerts with affected service lists delivered to Slack. The platform enforces license compliance scanning, detecting GPL/AGPL dependencies in production (critical legal risk in banking), flagging them for legal review. For SFC audits, Dependency-Track provides timestamped, versioned component inventories per service-machine-readable evidence that Addi maintains comprehensive software component tracking as mandated by CE 007/2018.

## 3. SBOM Lifecycle

```
Build Pipeline
      ↓
Trivy scans image -> generates CycloneDX SBOM
      ↓
Cosign attaches SBOM to OCI image artifact
      ↓
Upload SBOM to Dependency-Track
      ↓
  ┌─────────────────────────────────────────┐
  │                                         │
CVE published -> match against indexed SBOMs -> Slack alert (affected services)
  │                                         │
License scan -> detect GPL/AGPL -> flag for legal review
  │                                         │
Audit request -> query component inventory -> generate compliance report
  └─────────────────────────────────────────┘
```

## 4. SFC Mapping

SFC Circular Externa CE 007/2018 requires banking platforms to maintain "inventory of software components" as part of vulnerability management controls. SBOMs satisfy this requirement automatically: every build produces a machine-readable, cryptographically signed component inventory. By uploading SBOMs to Dependency-Track, Addi creates an audit trail proving component tracking is continuous, versioned per service, and timestamped. This evidence directly demonstrates CE 007/2018 compliance, streamlining SFC examinations and reducing audit friction through transparent, reproducible supply chain visibility.
